BrowserPod 2.0 is here with a much broader toolkit for developers.

With 2.0, BrowserPod adds a real bash shell, git, and core Linux utilities to the browser. That makes it much easier to bring familiar developer workflows into BrowserPod, from cloning repositories and running shell scripts to inspecting files and working through a real command line environment.

## What’s new in BrowserPod 2.0?

- Bash
- Git
- BusyBox with tools such as `cp`, `rm`, `ls`, `sed`, `grep`, and `vi`
- Expanded framework support across workflows such as Next.js, Nuxt, and Vue
- `fork`

### Remind me, what is BrowserPod?

BrowserPod is an in-browser code sandbox. It aims to implement the Linux syscall interface that many applications expect, so software can be compiled to WebAssembly with little to no code change, and multiple applications can interact with each other.

It supports outbound internet access for downloading packages or calling APIs, and inbound connections for exposing local development servers.

That makes it useful for safe in-browser agentic code execution, web-based IDEs and development environments, interactive docs and live demos, educational platforms, and other applications that benefit from sandboxed execution inside a web app.

### Git

We have compiled the official `git` CLI to WebAssembly and included it in the BrowserPod system image.

Repositories are the natural starting point for many developer workflows. With git available inside BrowserPod, you can clone a repository, inspect history, create commits, and push changes using the same commands developers already know.

Supply chain attacks and malicious repositories are an especially prescient concern. BrowserPod’s sandboxed environment gives developers a safer place to inspect unfamiliar code without exposing their local machine.

We expect this to be especially useful for web IDEs, feature branch previews, and AI agent sandboxes.

### Bash

Adding a real shell is one of the most important changes in BrowserPod 2.0.

Many non-trivial Linux applications depend on shell scripts for setup and orchestration. Wrapper scripts prepare the environment, forward arguments, and chain commands together. Without shell support, a lot of that logic has to be recreated elsewhere. With bash available inside the pod, those workflows become much easier to support directly.

An interactive shell also makes BrowserPod much easier to use. Instead of driving everything through a narrow API surface, users and agents can work through command line conventions they already understand.

### BusyBox

Bash on its own would not be very useful, which is why BrowserPod 2.0 also includes BusyBox. That gives the pod a practical set of Linux utilities such as `cp`, `rm`, `ls`, `sed`, `grep`, and `vi`, which makes shell workflows genuinely usable.

### Expanded framework support

BrowserPod 2.0 widens support across workflows built with frameworks such as Next.js, Nuxt, and Vue.

### Fork

Another major addition in BrowserPod 2.0 is support for `fork`.

`fork` is a system call that creates a new process by cloning the entire state of the caller process. It returns different values to the parent and the child, which then continue execution independently.

Supporting `fork()` in WebAssembly is difficult. WebAssembly does not provide a direct way to clone the call stack or resume execution from an arbitrary point.

We will publish a dedicated technical post on the implementation soon. In short, we made it work through a combination of compiler-injected instrumentation, stack unwinding via exceptions, and an embedded Wasm interpreter.

Once `fork` was in place, porting additional utilities to BrowserPod became much easier. That is one of the reasons this release expands the practical range of software the platform can support.
